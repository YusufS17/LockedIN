<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| App ↔ UserDefaults (local persistence) | Cosmetic identity (CharacterAppearance JSON + display name string) is written to and read back from on-device UserDefaults. The read path treats stored bytes as untrusted (could be stale/corrupt after a schema change). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03 | Denial of Service (crash) | `CharacterPersistence.load()` decoding stale/corrupt UserDefaults JSON | mitigate | `try?` on `JSONDecoder().decode` returns nil; AppStore falls back to `CharacterAppearance.default`; no force-unwrap (RESEARCH §Security V5) |
| T-02-04 | Information Disclosure | Persisted identity data | accept | Only a self-chosen display name + cosmetic appearance enums are stored; no PII, credentials, money, or device/app-usage data; UserDefaults un-encrypted is acceptable for non-sensitive cosmetic data (RESEARCH §Security V6) |
| T-02-05 | Information Disclosure | Logging of identity values | mitigate | Do not log display name or appearance to console/os_log in persistence or AppStore code (no `print(displayName)` etc.) — keeps identity data off any captured logs |
| T-02-SC | Tampering | npm/pip/cargo installs | accept | No package installs (CLAUDE.md forbids SPM; RESEARCH §Package Legitimacy Audit: not applicable). No supply-chain surface. |
</threat_model>
